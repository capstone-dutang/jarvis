# JARVIS 캡스톤 최종보고서

## 1. 프로젝트명

**JARVIS: AI 대화 기반 클라우드 장기기억 및 MCP 회상 시스템**

## 2. 프로젝트 개요

JARVIS는 AI 클라이언트가 사용자와 나눈 대화의 핵심 맥락을 클라우드에 저장하고, 이후 다른 세션이나 다른 AI가 MCP 도구를 통해 과거 맥락을 회상할 수 있도록 하는 장기기억 시스템이다. 사용자는 AI와 장기간 프로젝트를 진행하면서 매번 같은 설명을 반복해야 하는 문제를 겪는다. 특히 개발 프로젝트에서는 기술 선택 이유, 이전 오류 해결 과정, 기능 요구사항, 다음 작업 계획이 세션이 바뀔 때마다 사라지는 문제가 크다.

본 프로젝트는 이러한 문제를 해결하기 위해 AI 대화를 단순 로그로 저장하는 것이 아니라, AI가 다시 사용할 수 있는 구조화된 기억으로 변환하여 저장하는 것을 목표로 하였다. JARVIS는 대화 내용을 episode, turn, entity, fact, relation, fragment로 나누어 저장하고, 의미 검색, 전문 검색, 그래프 탐색을 결합하여 과거 맥락을 회상한다.

## 3. 개발 배경 및 필요성

최근 GPT, Claude, Codex 등 다양한 AI 도구가 개발과 문서 작성 과정에서 널리 사용되고 있다. 그러나 대부분의 AI 도구는 세션 단위로 맥락이 제한된다. 하나의 세션에서는 높은 생산성을 제공하지만, 세션이 종료되거나 다른 AI 클라이언트로 이동하면 이전 대화의 맥락이 유지되지 않는다. 사용자는 프로젝트 설명, 기술 선택 이유, 이전 오류 상황, 앞으로 해야 할 일을 반복해서 설명해야 한다.

이 문제는 단순히 대화 기록을 파일로 저장한다고 해결되지 않는다. 긴 로그를 사람이 다시 읽는 것은 시간이 많이 걸리고, AI에게 전체 로그를 다시 입력하는 것은 토큰 비용과 컨텍스트 길이 측면에서 비효율적이다. 따라서 필요한 것은 사람이 보관하는 로그가 아니라, AI가 직접 검색하고 이해할 수 있는 장기기억 구조이다.

JARVIS는 이러한 문제를 해결하기 위해 설계되었다. 핵심 아이디어는 대화 중인 AI가 이미 현재 맥락을 알고 있으므로, 해당 AI가 대화의 핵심 내용을 요약하고 구조화하여 기억 서버에 저장하도록 하는 것이다. 서버는 별도의 LLM 분석을 수행하지 않고, 저장, 검증, 색인, 검색을 담당한다. 이 구조는 비용을 줄이면서 여러 AI 클라이언트가 같은 기억 저장소를 공유할 수 있게 한다.

## 4. 프로젝트 목표

본 프로젝트의 목표는 다음과 같다.

1. AI 대화 내용을 워크스페이스 단위로 저장할 수 있는 서버를 구현한다.
2. 저장된 대화를 단순 텍스트가 아니라 entity, fact, relation 중심의 지식 구조로 관리한다.
3. MCP 도구를 통해 AI 클라이언트가 기억을 저장하고 회상할 수 있도록 한다.
4. 의미 검색, 키워드 검색, 그래프 탐색을 결합한 하이브리드 검색 구조를 구현한다.
5. 웹 UI를 통해 저장된 일기와 주요 주제를 사람이 확인할 수 있도록 한다.
6. Docker Compose와 GitHub Release를 통해 실행 및 제출 가능한 최종 결과물을 제공한다.

## 5. 시스템 아키텍처

JARVIS는 크게 FastAPI 백엔드, PostgreSQL 데이터베이스, MCP 어댑터, 웹 UI로 구성된다.

FastAPI 백엔드는 REST API와 MCP endpoint를 제공한다. REST API는 워크스페이스, 사용자, 메모리 저장, 검색, 브리핑, episode 조회 등의 기능을 담당한다. MCP endpoint는 AI 클라이언트가 직접 호출하는 도구 인터페이스로 사용된다.

PostgreSQL은 JARVIS의 핵심 저장소이다. 일반 관계형 데이터뿐 아니라 pgvector를 이용한 벡터 검색, PGroonga를 이용한 한국어 전문 검색, recursive query를 이용한 그래프 탐색을 함께 처리한다. 이를 통해 별도의 벡터 DB나 그래프 DB 없이도 캡스톤 MVP에 필요한 검색 기능을 통합할 수 있었다.

웹 UI는 FastAPI에서 정적 파일로 서빙되는 단일 페이지 애플리케이션이다. 사용자는 브라우저에서 날짜별 일기, 주요 주제, 검색 결과, 엔티티 정보를 확인할 수 있다.

## 6. 주요 데이터 모델

JARVIS의 핵심 데이터 모델은 다음과 같다.

- **Workspace**: 프로젝트 또는 작업 단위의 기억 공간이다. 서로 다른 프로젝트의 기억이 섞이지 않도록 분리한다.
- **Episode**: 하나의 대화 또는 하루 작업 단위의 기록이다. summary, diary_entry, human_summary 등을 포함한다.
- **Turn**: 사용자와 AI의 발화 단위이다. 대화의 흐름을 복원하는 데 사용된다.
- **Entity**: 프로젝트, 기술, 사람, 기능, 개념 등 기억의 중심이 되는 대상이다.
- **KnowledgeFact**: 특정 entity에 대한 구조화된 사실이다. predicate와 object_value를 통해 사실을 표현한다.
- **EntityRelation**: entity 간 관계를 나타낸다. 그래프 탐색과 관련 맥락 회상에 사용된다.
- **Fragment**: 의미 검색을 위한 자연어 조각이다. 구조화된 fact만으로 부족한 대화 맥락을 검색하기 위해 사용된다.

이 모델은 대화를 단순한 문자열로 저장하는 방식보다 검색과 회상에 유리하다. 예를 들어 “PostgreSQL을 선택한 이유”를 검색할 때, PostgreSQL이라는 entity와 관련된 fact, relation, fragment를 함께 탐색할 수 있다.

## 7. MCP 도구 설계

JARVIS는 MCP 도구를 통해 AI 클라이언트와 연결된다. 주요 도구는 다음과 같다.

- `jarvis_initialize_memory`: 세션 시작 시 워크스페이스의 최근 맥락과 사용 가능한 명령을 불러온다.
- `jarvis_log_diary`: 현재 대화를 일기 형태로 저장한다.
- `jarvis_recall_memory`: 자연어 질문을 기반으로 과거 기억을 검색한다.
- `jarvis_brief_me`: 현재 워크스페이스 또는 전체 워크스페이스의 최근 상태를 브리핑한다.
- `jarvis_explore_topic`: 특정 주제 주변의 엔티티와 관계를 탐색한다.
- `jarvis_search_passages`: 구조화된 fact만으로 부족할 때 대화 passage를 검색한다.
- `jarvis_get_episode_excerpt`: 특정 episode 안에서 질문과 관련된 부분을 발췌한다.
- `jarvis_follow_relation`: 특정 entity와 연결된 주변 entity를 따라간다.

도구 이름에는 `jarvis_` 접두사를 붙여 다른 MCP 도구와 구분하였다. 또한 각 도구 설명에는 사용해야 하는 상황과 사용하지 말아야 하는 상황을 명확히 작성하여 AI가 적절한 도구를 선택할 수 있도록 하였다.

## 8. 기억 저장 방식

초기에는 raw transcript를 그대로 저장하는 방식을 고려하였다. 그러나 대화 원문에는 긴 도구 실행 결과, 파일 읽기 결과, 반복 로그, 코드 블록이 많이 포함되어 있어 그대로 저장하면 비효율적이었다. 또한 일부 AI 환경에서는 로컬 transcript 파일에 접근할 수 없기 때문에 raw 파일 업로드 방식은 범용성이 떨어졌다.

이를 해결하기 위해 JARVIS는 “AI가 쓰는 일기” 모델을 채택하였다. 대화 중인 AI가 현재 컨텍스트를 바탕으로 사용자 발화, assistant 요약, 전체 summary, diary_entry, human_summary, keywords, entities, facts, relations를 재구성하여 `jarvis_log_diary`로 저장한다. 이 방식은 원문 전체를 저장하지 않아도 핵심 맥락을 유지할 수 있고, 다른 AI 클라이언트에서도 동일한 방식으로 사용할 수 있다.

## 9. 검색 및 회상 구조

JARVIS의 검색 구조는 세 가지 방식의 결합으로 구성된다.

첫째, pgvector 기반 의미 검색이다. 사용자의 질문과 의미적으로 유사한 fragment나 fact를 찾는다. 표현이 다르더라도 비슷한 의미의 대화를 찾는 데 유리하다.

둘째, PGroonga 기반 전문 검색이다. 기술명, 프로젝트명, 함수명, 고유명사처럼 정확한 키워드가 중요한 경우에 유리하다. 특히 한국어 검색을 지원하기 위해 PGroonga를 사용하였다.

셋째, entity relation 기반 그래프 탐색이다. 특정 entity와 연결된 주변 entity를 따라가며 관련 맥락을 찾는다. 예를 들어 MCP를 검색하면 MCP endpoint, OAuth, Streamable HTTP, tool invocation 같은 관련 개념으로 확장할 수 있다.

이 세 가지 검색 결과를 결합하여 AI에게 필요한 맥락을 제공한다. 이를 통해 JARVIS는 단순한 문자열 검색보다 더 풍부한 회상 결과를 제공할 수 있다.

## 10. 웹 UI

웹 UI는 JARVIS에 저장된 기억을 사람이 확인하기 위한 화면이다. 주요 기능은 날짜별 episode 확인, 최근 작업 브리핑, 주요 주제 확인, 검색 결과 탐색, entity 정보 확인이다. 캡스톤 MVP에서는 복잡한 프론트엔드 프레임워크보다 배포와 유지가 쉬운 단일 파일 SPA 구조를 선택하였다.

웹 UI의 목적은 모든 기능을 완벽히 제공하는 것이 아니라, 저장된 기억이 실제로 존재하고 날짜와 주제별로 확인 가능하다는 점을 보여주는 것이다. 이를 통해 MCP 도구만으로 보이지 않는 저장 결과를 사람이 직접 검증할 수 있다.

## 11. 구현 기술

본 프로젝트에서 사용한 주요 기술은 다음과 같다.

- Python 3.11
- FastAPI
- SQLAlchemy
- Alembic
- PostgreSQL 16
- pgvector
- PGroonga
- MCP Python SDK
- Docker Compose
- 단일 파일 웹 UI
- GitHub Release 기반 제출 버전 관리

FastAPI는 비동기 API 서버를 빠르게 구성할 수 있고, MCP endpoint와 REST API를 함께 제공하기에 적합하였다. PostgreSQL은 관계형 데이터, 벡터 검색, 전문 검색을 한 저장소 안에서 처리할 수 있어 MVP 구현에 적합하였다. Docker Compose는 개발 환경 재현성을 높이기 위해 사용하였다.

## 12. 테스트 및 검증

테스트와 검증은 다음 흐름을 중심으로 진행하였다.

1. Docker Compose를 이용해 서버와 데이터베이스가 정상 실행되는지 확인하였다.
2. `/health` endpoint를 통해 서버 상태를 점검하였다.
3. 워크스페이스 생성 및 조회 기능을 확인하였다.
4. `jarvis_log_diary`를 통해 대화 기록이 저장되는지 확인하였다.
5. `jarvis_initialize_memory`로 워크스페이스 상태가 반환되는지 확인하였다.
6. `jarvis_recall_memory`로 저장된 기억을 회상할 수 있는지 확인하였다.
7. passage 검색과 episode 발췌 기능을 통해 구조화된 fact 외의 대화 맥락도 찾을 수 있는지 확인하였다.
8. 웹 UI에서 저장된 episode와 주요 주제가 표시되는지 확인하였다.

또한 개발 과정에서 연구 노트와 검증 문서를 작성하여 설계 변경 이유와 구현 결과를 기록하였다.

## 14. 기대 효과

JARVIS를 사용하면 사용자는 AI와 장기간 작업할 때 반복 설명을 줄일 수 있다. AI는 과거 대화에서 어떤 결정을 했는지, 어떤 문제가 있었는지, 다음에 무엇을 해야 하는지 회상할 수 있다. 이는 개인 개발 프로젝트뿐 아니라 팀 프로젝트 온보딩, 연구 기록 관리, 문서 작성, 기획 작업에도 활용 가능하다.

또한 JARVIS는 특정 AI 플랫폼에 종속되지 않는다. MCP를 지원하는 여러 AI 클라이언트가 같은 기억 저장소를 사용할 수 있으므로, 사용자는 GPT, Claude, Codex 등 도구를 바꾸더라도 프로젝트 맥락을 유지할 수 있다.

## 15. 한계점

현재 구현은 캡스톤 MVP 수준이므로 몇 가지 한계가 있다. 첫째, 인증과 권한 관리가 상용 서비스 수준으로 완성된 것은 아니다. 둘째, AI가 항상 적절한 시점에 기억 저장 도구를 호출하도록 보장하기 어렵다. 셋째, 웹 UI는 핵심 확인 기능 중심으로 구현되어 고급 그래프 시각화나 협업 기능은 제한적이다. 넷째, 대규모 사용자와 대량 데이터 환경에서의 성능 검증은 추가로 필요하다.

## 16. 향후 개선 방향

향후에는 OAuth 기반 인증과 사용자 권한 관리를 강화할 수 있다. 팀 단위 워크스페이스 초대, 읽기/쓰기 권한 분리, 민감 정보 필터링 기능도 추가할 수 있다. 또한 기억 중복 제거, 모순 감지, 변경 이력 시각화 기능을 고도화하면 장기 프로젝트에서 더 정확한 회상이 가능해진다.

웹 UI 측면에서는 entity graph 시각화, 날짜별 타임라인, 주제별 필터, 검색 결과 비교 기능을 추가할 수 있다. 배포 측면에서는 CI/CD와 자동 마이그레이션, 모니터링을 추가하여 운영 안정성을 높일 수 있다.

## 17. 결론

JARVIS는 AI 세션 간 맥락 단절 문제를 해결하기 위해 설계된 클라우드 기반 장기기억 시스템이다. 본 프로젝트에서는 MCP 연동, 일기 기반 대화 저장, 구조화된 지식 모델, 하이브리드 검색, 웹 UI를 구현하여 AI가 과거 작업 맥락을 회상하고 다음 세션에서 이어서 작업할 수 있는 흐름을 완성하였다.

최종적으로 JARVIS는 “AI가 나와 함께 작업한 기록을 기억하고, 다음 세션에서 이어서 도와주는 시스템”이라는 목표를 MVP 형태로 달성하였다. 캡스톤 결과물은 GitHub Release로 고정되어 제출 가능하며, 향후 인증, 협업, 시각화, 운영 안정성을 보강하면 실제 장기 프로젝트 관리 도구로 확장할 수 있다.

# 13주차 주간보고서

## 금주예정작업:

이번 주에는 캡스톤 최종 제출을 위한 통합 작업을 진행하는 것을 목표로 하였다. 지금까지 구현한 MCP 도구, 백엔드 API, 데이터베이스 스키마, 검색 파이프라인, 웹 UI를 최종 상태로 정리하고, 실행 방법과 제출용 GitHub Release를 준비할 예정이었다. 또한 최종보고서와 주간보고서를 정리하여 프로젝트의 개발 과정과 결과물이 평가자에게 명확히 전달되도록 하는 것이 주요 작업이었다.

구체적으로는 README를 최신 구조에 맞게 갱신하고, Docker Compose 기반 실행 방법을 정리하며, 배포 환경에서 MCP endpoint가 동작하도록 설정을 확인하기로 하였다. 최종 소스코드는 특정 커밋과 Release 태그로 고정하여 제출 가능하도록 만들 계획이었다.

## 금주 수행결과 및 문제점 해결방법:

금주에는 최종 통합 작업을 수행하였다. MCP 도구 목록과 README를 정리하고, JARVIS의 핵심 개념을 “AI가 매일 일기를 쓰는 클라우드 기억 저장소”로 설명하도록 문서를 재작성하였다. Dockerfile에는 임베딩 모델 사전 탑재와 실행 환경 구성을 반영하였고, Railway와 같은 배포 환경에서 서버가 지정된 포트로 실행될 수 있도록 설정을 보완하였다. 또한 프록시 환경에서 MCP endpoint가 정상 동작하도록 관련 설정을 조정하였다.

문제점은 최종 제출용 소스코드를 어떤 형태로 고정할지였다. 단순 GitHub main branch 주소를 제출하면 이후 변경 사항과 제출 시점의 코드가 섞일 수 있다. 이를 해결하기 위해 GitHub Release를 생성하여 제출 버전을 고정하였다. 최종 제출용 Release는 `v1.0.1-capstone`으로 생성하였으며, 평가자는 해당 Release 주소를 통해 제출 시점의 소스코드를 확인할 수 있다.

## 지도교수 면담 및 토론결과:

지도교수와의 최종 면담에서는 결과물 제출 방식과 발표 시 강조할 내용을 점검하였다. 교수님은 최종 발표에서 기술 요소를 모두 나열하기보다, 문제 정의에서 출발하여 “AI 세션 간 기억 단절을 어떻게 해결했는가”를 중심으로 설명하는 것이 좋다고 조언하였다. 또한 Release 주소를 제출하면 평가 시점의 코드가 명확히 고정되므로, 최종 산출물 관리 측면에서 적절하다는 의견을 받았다.

토론 결과 발표 구성은 문제 정의, 시스템 구조, MCP 저장 및 회상 흐름, 웹 UI, 기대 효과, 한계 및 향후 개선 방향 순서로 정리하기로 하였다. 시연은 세션 초기화, 일기 저장, 과거 맥락 회상 흐름을 중심으로 구성하기로 하였다.

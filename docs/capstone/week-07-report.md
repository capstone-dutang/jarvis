# 7주차 주간보고서

## 금주예정작업:

이번 주에는 JARVIS의 방향을 로컬 개인 도구에서 클라우드 기반 컨텍스트 서버로 확장하는 작업을 진행할 예정이었다. 프로젝트 초기에는 로컬 환경에서 동작하는 개인 기억 시스템을 중심으로 생각했으나, 실제 문제는 여러 AI 클라이언트와 여러 세션 사이에서 맥락이 이어지지 않는다는 점이므로 원격 서버 형태가 더 적합하다고 판단하였다. 이에 따라 MCP를 통해 외부 AI 클라이언트가 접근할 수 있는 구조를 검토하기로 하였다.

또한 JARVIS가 서버에서 모든 대화를 분석하는 방식과, 대화 중인 AI가 스스로 핵심 내용을 정리해서 서버에 보내는 방식 중 어느 쪽이 더 적합한지 비교하기로 하였다. 서버 비용, 구현 난이도, 클라이언트 호환성을 기준으로 구조를 결정하는 것이 주요 예정 작업이었다.

## 금주 수행결과 및 문제점 해결방법:

금주에는 JARVIS를 “AI 클라이언트들이 공유하는 클라우드 기억 서버”로 재정의하였다. 사용자가 GPT에서 논의한 내용을 Claude 또는 Codex에서 이어받을 수 있도록, 기억 저장과 회상을 MCP 도구 형태로 제공하는 방향을 확정하였다. 이 구조에서는 AI 클라이언트가 MCP 도구를 호출하여 대화 내용을 저장하고, 이후 다른 세션에서 같은 워크스페이스의 기억을 회상할 수 있다.

중요한 문제는 서버가 대화를 직접 분석하려면 LLM 호출 비용이 계속 발생한다는 점이었다. 이를 해결하기 위해 “클라이언트가 구조화하고 서버는 검증·저장·검색한다”는 원칙을 세웠다. 대화 중인 AI는 이미 전체 맥락을 알고 있으므로, 해당 AI가 summary, turns, entities, facts, relations를 구성해서 JARVIS에 전송하고, 서버는 이를 저장하고 검색하는 역할에 집중한다. 이 방식은 서버 비용을 줄이고 여러 AI 클라이언트에 적용하기 쉽다는 장점이 있다.

## 지도교수 면담 및 토론결과:

지도교수 면담에서는 클라우드 서버로 방향을 확장할 경우 구현 범위가 너무 커지지 않도록 주의해야 한다는 의견이 있었다. 교수님은 최종 결과물에서 원격 배포나 인증 전체를 완벽히 구현하기보다, MCP endpoint를 통해 AI가 기억을 저장하고 회상하는 핵심 흐름을 보여주는 것이 중요하다고 조언하였다.

또한 “서버가 LLM을 사용하지 않는 구조”가 프로젝트의 차별점이 될 수 있다는 피드백을 받았다. 이에 따라 최종보고서에는 기존 기억 시스템과 달리 JARVIS가 클라이언트 AI의 컨텍스트를 활용해 비용을 줄이는 구조임을 강조하기로 하였다.

# 8주차 주간보고서

## 금주예정작업:

이번 주에는 MCP 기반 기억 저장이 실제로 안정적으로 동작하기 위해 필요한 캡처 전략을 연구하는 것을 목표로 하였다. AI가 항상 적절한 시점에 저장 도구를 호출한다는 보장이 없기 때문에, 기억 누락을 줄이기 위한 보조 경로를 설계할 필요가 있었다. 특히 긴 대화에서 앞부분 맥락이 사라지거나, 사용자가 명시적으로 저장을 요청하지 않은 경우에도 중요한 결정이 누락되지 않도록 하는 방안을 검토하였다.

또한 MCP 도구 설명, 세션 시작 초기화, 저장 명령, 회상 명령의 자연어 트리거를 정리하고자 하였다. 최종적으로는 JARVIS가 AI 클라이언트에게 어떤 도구를 언제 호출해야 하는지 명확히 알려줄 수 있는 도구 설계 원칙을 만드는 것이 목표였다.

## 금주 수행결과 및 문제점 해결방법:

금주에는 기억 캡처 아키텍처를 3경로로 정리하였다. 첫 번째는 대화 중간에 AI가 핵심 내용을 증분 저장하는 경로이다. 두 번째는 대화 원본 또는 episode 단위의 내용을 보존하여 나중에 재처리할 수 있도록 하는 경로이다. 세 번째는 세션 종료 후 미정리 대화를 감지하고 다시 저장을 유도하는 복구 경로이다. 이 구조를 통해 MCP 도구 호출이 누락되더라도 완전히 기억이 사라지지 않도록 설계하였다.

문제점은 AI 클라이언트마다 MCP 도구 호출 방식과 적극성이 다르다는 점이었다. 어떤 클라이언트는 도구 설명을 잘 따르지만, 어떤 환경에서는 사용자가 명시적으로 요청해야 도구를 호출할 가능성이 높았다. 이를 해결하기 위해 도구 이름에 `jarvis_` 접두사를 붙이고, 각 도구 설명에 “언제 사용해야 하는지”를 명확히 작성하는 원칙을 세웠다. 또한 세션 시작 시 `jarvis_initialize_memory`를 통해 워크스페이스 상태와 사용 가능한 명령을 안내하는 방식을 도입하였다.

## 지도교수 면담 및 토론결과:

지도교수와의 토론에서는 “AI가 저장 도구를 호출하지 않으면 어떻게 되는가”라는 질문이 있었다. 교수님은 이 부분이 실제 시스템의 신뢰성과 직접 연결되므로, 최종보고서에서 실패 시나리오와 대응 방식을 반드시 설명해야 한다고 조언하였다. 이에 따라 단일 저장 경로에 의존하지 않고 증분 저장, 원본 보존, 복구 경로를 함께 사용하는 구조를 보고서에 포함하기로 하였다.

또한 시연에서는 모든 실패 대응을 보여주기보다, 기본 흐름과 함께 설계상 어떤 보완책을 두었는지 설명하면 충분하다는 의견을 받았다.
